# Portable Storage Schema Mapping

This document defines how the Story Bible storage contract maps PostgreSQL/Supabase concepts to local SQLite without leaking provider-specific behavior into services.

| PostgreSQL / Supabase | SQLite Mapping | Notes |
| --- | --- | --- |
| UUID | `TEXT` | IDs are generated by services or adapter helpers and remain stable strings. |
| JSONB | `TEXT` containing stable JSON | Values are serialized with `JSON.stringify`, parsed on read, and schema-validated by service contracts. |
| TIMESTAMPTZ | `TEXT` UTC ISO-8601 | Store `new Date().toISOString()`. |
| BOOLEAN | `INTEGER` 0/1 | Adapter converts to booleans in returned JSON when needed. |
| ENUM | `TEXT` with CHECK where stable | Early L0B.1 keeps most enums in JSON until contract locks. |
| ARRAY | JSON `TEXT` or relation table | Relation tables are used for candidate/source/conflict/canonical source links. |
| BIGINT | `INTEGER` | SQLite stores signed 64-bit integers. |
| Partial index | SQLite partial index where required | L0B.1 uses simple indexes first. |
| Advisory lock | Transaction/lock table | L0B.1 uses `BEGIN IMMEDIATE`; future slices may add lock rows. |

## Stable JSON

Provider-specific rows are stored as JSON text. The adapter is responsible for:

- Serialization.
- Parsing.
- Validation by public storage contract.
- Error mapping to storage errors.

Services must not read or write SQLite table details directly.

## Canonical Entity Mapping

Supabase can expose multiple canonical entity tables. SQLite L0B uses one portable table:

`canonical_entities(project_id, entity_type, entity_id, active, version_number, row_json)`

This keeps project isolation and entity polymorphism in one local file while allowing the adapter to return the same contract objects.

## Source Identity

Source natural keys use the existing `source-natural-key-v1` hash. SQLite enforces `UNIQUE(project_id, natural_key_hash)` so retry-safe source dedup works without cloud state.

## Project Isolation

L0B uses one database file per project: `<projectExportId>.novel.sqlite`. Every row still carries `project_id`, and relations cannot intentionally cross project boundaries. Shared databases are not used for user projects.
